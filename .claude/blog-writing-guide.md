# 블로그 글쓰기 스타일 가이드

이 문서는 velog 블로그 글 작성 시 일관된 스타일을 유지하기 위한 가이드입니다.

---

## 1. 제목 스타일

### 선호하는 패턴

- **질문형**: 독자의 호기심 유발
  - "~하고 있나요?", "~일까?", "~해야할까?"
  - 예: "ErrorBoundary를 잘 사용하고 있나요?"
  - 예: "매일 아침 스탠드업, 직접 쓰고 계신가요?"

- **키워드 강조형**: 핵심 기술/개념 명시
  - 예: "프론트엔드에서 도메인 모델링이 필요한 이유"
  - 예: "Todo 앱에 TDD 적용해보기"

- **삽질기/회고형**: 실패와 교훈을 솔직하게
  - "~삽질기", "(그리고 ~한 이유)"
  - 예: "Docker + EKS 삽질기 (그리고 EC2로 돌아온 이유)"
  - 예: "3일 삽질 끝에 깨달은 것"

### 피해야 할 패턴
- 너무 길거나 모호한 제목
- 클릭베이트성 과장 표현

---

## 2. 부제목 (short_description)

- 한 줄로 글의 핵심 또는 독자 공감 포인트 요약
- 약간의 위트나 개인적 느낌 포함 가능
- **velog에서 잘리지 않도록 짧게** (50자 이내 권장)

```
제목: 매일 아침 스탠드업, 직접 쓰고 계신가요?
부제: 클로드와 함께 편하게 일해보자

제목: E2E 테스트 인프라, Docker + EKS 삽질기
부제: 3일간의 EKS 삽질 끝에 EC2로 2시간 만에 해결한 이야기

제목: 슬기로운 인턴 생활 - (5)
부제: 45일이 지났다....
```

## 3. 도입부 작성법

### 방식 1: 공감 유발형
독자가 겪었을 법한 상황으로 시작

```markdown
## 🤔 매일 반복되는 업무 루틴

팀에서 스탠드업 메시지 작성을 기본적으로 하고 있고...
그래서 저의 하루는 비슷한 패턴으로 시작하고 끝납니다.

> "이거 자동화하면 안 될까?"

그래서 만들었습니다.
```

### 방식 2: 결론 먼저 보여주기 (삽질기에 효과적)
인용구로 결론을 먼저 제시하여 호기심 유발

```markdown
> "제대로 해보자"며 Docker + EKS로 시작했습니다.
> 3일 만에 포기하고 EC2로 돌아왔습니다.
> 그 삽질의 기록과, 오버엔지니어링에서 배운 것들.
```

**효과**: 독자가 "왜 포기했지?" 궁금해서 끝까지 읽게 됨

### 핵심 요소
- 1인칭 서술 ("저는", "제가")
- 구체적인 상황 묘사
- 인용구(`>`)로 핵심 질문/생각 강조

## 4. 섹션 헤딩 스타일

### 이모지 + 제목 형식
각 섹션은 이모지로 시작하여 시각적 구분과 톤을 설정합니다.

```markdown
## 🎯 시작 / 목표
## 🐳 1일차 / 첫 번째 시도
## ☸️ 2일차 / 두 번째 시도
## 🤯 3일차 / 현타 / 전환점
## 🔄 방향 전환
## 🤔 왜 필요한가 / 고민
## 💻 구현 / 코드
## 📌 핵심 포인트 / 정리
## 🔧 해결 방법 / 도구
## ⚠️ 주의사항 / 삽질 기록
## 💡 오버엔지니어링의 징후들 / 배운 것들
## 🛠️ 현재 아키텍처
## 🚀 자동화 로드맵 / 다음 단계
## 📚 배운 것들 / 교훈
## 🎉 결과 / 마무리
```

### 하위 섹션
이모지 없이 일반 헤딩 사용:
```markdown
### 선호하는 패턴
### Phase 1: 첫 번째 단계
### 1. "나중에 필요할 것 같아서"
```

## 5. 톤 & 어조

### 기본 톤
- **편안한 구어체**: 딱딱하지 않게
- **독자와 대화하는 느낌**: 질문 던지기
- **솔직한 실패 공유**: 잘못된 선택을 인정

### 자주 사용하는 표현

| 상황 | 표현 예시 |
|------|----------|
| 실수 인정 | "첫 번째 실수는 여기서 시작됐습니다" |
| 현타 | "잠깐, 내가 지금 뭘 하고 있는 거지?" |
| 방향 전환 | "점심 먹고 결정했습니다" |
| 의견 제시 | "~라고 생각한다", "~에 아주 동의한다" |
| 질문 유도 | "어떤 게 더 비용이 적을까요?" |
| 경험 공유 | "~하고 있었습니다" |
| 공감 유도 | "~인 건 저만 그런 게 아니겠죠?" |
| 강조 | "**핵심은**", "**즉,**" |
| 자조 | "~~(아마도)~~" |

### 피해야 할 톤
- 과도하게 격식체 ("~하였습니다", "~인 것이다")
- 권위적 어조 ("반드시 ~해야 한다")
- 실패를 숨기거나 포장하기

## 6. 글 유형별 구조

### 삽질기/회고 글 (오버엔지니어링, 실패 경험)

**핵심**: 시간순 스토리텔링 + 솔직한 실패 공유

```
1. 인용구 도입부 (결론 먼저)
   > "3일 삽질 끝에 포기했습니다"

2. 🎯 시작: 처음 계획/동기
   - 요구사항은 단순했는데...
   - "이왕 하는 거 제대로 하자"

3. 🐳 1일차: 첫 번째 시도
   - 구체적인 작업 내용
   - 마주친 문제들

4. ☸️ 2일차: 계속되는 시도
   - 진행 상황
   - 쌓이는 문제들

5. 🤯 3일차: 현타
   - 체크리스트로 현재 상태 정리 (✅❌)
   - "내가 지금 뭘 하고 있는 거지?"
   - 본질 vs 현실 비교

6. 🔄 방향 전환
   - 결정의 순간
   - 새로운 접근법
   - 타임라인 표로 정리

7. 💡 오버엔지니어링의 징후들
   - 3가지 정도로 정리
   - 각각 구체적 예시 포함

8. 🛠️ 지금의 아키텍처
   - 간단한 다이어그램
   - "단순합니다. 그리고 잘 돌아갑니다."

9. 🚀 자동화 로드맵 (선택)
   - Phase 1, 2, 3...
   - "필요하면 그때"

10. 📚 배운 것들
    - YAGNI, 복잡함은 비용 등
    - 번호 매겨서 정리

11. 마무리
    - 핵심 메시지 한 줄
    - TL;DR
```

### 기술 실용 글 (자동화, 도구 소개)
```
1. 🤔 문제 상황 / 배경
2. ❗️ 해결책 소개
3. 🔧 구현 상세 (코드, 설정)
4. ⚠️ 삽질 기록 (선택)
5. 🎯 적용 방법
6. 🎉 결과 / 효과
7. 마무리 (GitHub 링크 등)
```

### 개념 설명 글 (DDD, 패턴)
```
1. ❓ 개념 정의
2. 📌 왜 필요한가 (목표)
3. ❗️ 적용 필요성 (구체적 이유들)
4. 코드 예시 / 실제 적용
5. 이점 정리
6. 📄 결론
```

### 기술 심층 분석 글 (ErrorBoundary)
```
1. ❗️ 주제 도입 (현재 방식의 문제)
2. 기존 방식 vs 새로운 방식 비교
3. 🤔 왜 사용해야 하는가
4. 어떻게 사용하면 좋을까
5. 주의사항 / 안티패턴
6. 참고 자료 링크
```

### 회고/경험 글 (인턴 생활)
```
1. ❗️ 시간/상황 도입
2. 💻 업무/활동 내용
3. 🍲 에피소드 (선택)
4. ❓ 느낀 점 / 회고
5. 사진으로 마무리 (선택)
```

## 8. 표(Table) 활용

### 타임라인 표
```markdown
| 시간 | 작업 | 상태 |
|------|------|------|
| 14:00 | EC2 인스턴스 생성 | ✅ |
| 14:15 | Node.js, pnpm 설치 | ✅ |
| 16:00 | **첫 E2E 테스트 성공** 🎉 | ✅ |
```

### Before vs After 비교
```markdown
| 방식 | 구축 시간 | 복잡도 |
|------|----------|--------|
| Docker + EKS | 1~2주 | 높음 |
| EC2 직접 구성 | **반나절** | 낮음 |
```

### 용도
- 옵션/기능 비교
- 타임라인 정리
- Before/After 대비
- 장단점 정리

## 9. 강조 & 포맷팅

### 인라인 강조
- **굵게**: 핵심 개념, 중요 단어, 숫자 강조
- `백틱`: 코드, 함수명, 파일명
- ~~취소선~~: 자조적 유머 ("~~아마도~~")

### 인용구 활용

**핵심 질문/깨달음**:
```markdown
> "잠깐, 내가 지금 뭘 하고 있는 거지?"
```

**도입부 요약** (삽질기):
```markdown
> "제대로 해보자"며 Docker + EKS로 시작했습니다.
> 3일 만에 포기하고 EC2로 돌아왔습니다.
```

### 리스트
- 순서 있는 항목: 숫자 리스트
- 나열 항목: 불릿 리스트
- 깊이는 2단계까지만
- 화살표(`→`)로 인과관계 표시

## 10. 결론 & TL;DR

### 마무리 구조
1. 핵심 메시지 (굵게 강조)
2. 열린 결론 또는 자조적 유머
3. TL;DR (선택, 삽질기에 효과적)

### 예시 (삽질기)
```markdown
## 마무리

3일간의 삽질 끝에 깨달았습니다.

**좋은 엔지니어링은 "최신 기술을 쓰는 것"이 아니라
"문제에 맞는 해결책을 찾는 것"입니다.**

나중에 정말 EKS가 필요해지면, 그때 전환하면 됩니다.
그때쯤이면 저도 Chromium 컨테이너 이슈를 해결할 수 있겠죠. ~~(아마도)~~

---

**TL;DR:**
- Docker + EKS로 3일 삽질
- EC2로 2시간 만에 해결
- 오버엔지니어링 주의
- 작동하는 게 먼저, 확장은 나중에
```

## 11. 태그 선택

### 기본 규칙
- 2~4개 태그 사용
- 주요 기술/개념 우선
- 기존에 사용한 태그 재활용

### 자주 사용하는 태그
| 카테고리 | 태그 |
|----------|------|
| 언어/프레임워크 | JavaScript, React, typescript |
| 방법론 | TDD, DDD, 함수형 프로그래밍 |
| 인프라 | AWS, Docker, E2E, Playwright |
| 주제 | 아키텍처, 리팩토링, 테스트, 자동화, 오버엔지니어링 |
| 경험 | 팀 프로젝트, 인턴, 이마고웍스 |
